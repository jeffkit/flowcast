# 2026-07-07

## 架构 review：flowx Executor 层能否用 agentproc 协议/ hub profile 实现

**结论：不建议替换，建议"组合"而非"取代"。**

- flowx 现有 7 个原生 adapter（claude/cursor/gemini/codex/aider/recursive/agy），直接 spawn CLI、解析各 CLI 原生输出、抽取富 `_meta`（tokens/model/exitCode/finishReason/budgetExceeded/panicked/transcriptMessages）。额外能力：provider 注入+fallback、schema 强约束(runStructured)、rate-limiter 持久化冷却、runAgentChain CLI 回退、SAFE_OPTS/EXTRA_ARGS/isSafePath 防注入、HITL/concurrency、onData 流式。
- agentproc hub 有 ~14 个 profile（含 deepseek/opencode/kimi/qwen/pi/codebuddy 等长尾），每个 bridge 把 CLI 原生输出翻译成 AGENT_SESSION/AGENT_PARTIAL/AGENT_ERROR 哨兵行。runner 输出只有 `{reply, sessionId, error, exitCode, timedOut}`——**丢失所有富 _meta**。

**直接替换的硬伤：**
1. 富 _meta 丢失（recursive 的 panicked/budgetExceeded 是 throwOnCritical→FlowcastError→fallback 的信号源，agentproc bridge 已消费掉不转发）。
2. 多一次进程跳转（`python3 bridge.py` 再 spawn CLI），额外失败面，流式精细控制退化。
3. provider 注入模型不匹配（flowx 的 resolveProvider/applyProvider vs agentproc 的 env/env_allowlist），且丢失 provider fallback。
4. 安全模型反转：flowx executor 是"防 LLM 注入配置"的守门员；agentproc runner 信任 profile 作者。
5. 丢失 flowx 专属能力（schema/rate-limit/fallback/concurrency/HITL）——这些要在 agentproc 之上重建，等于又把 executor 层造回来。
6. agentproc hub 自身有 parity debt（HANDOFF 记 recursive/bridge.js 不读 RECURSIVE_* env）。

**推荐路径（待 Jeff 拍板）：** 保留核心 7 adapter；新增一个泛型 `agentproc` executor 类型，仅用于 flowx 未原生支持的长尾 CLI（deepseek/kimi/qwen 等），把 `RunResult` 映射成 `makeAgentResult`；可选把富 telemetry 反向贡献回 agentproc 作为可选 onMeta 通道。

## 补充（同日）：ACP 协议可行性 — 比 agentproc 更合适

- **2026 现状**：ACP 由 Zed 2025.08 发布，JSON-RPC 2.0 over stdio（本地）/HTTP+WS（远程）。截至 2026 已有 30+ agent、20+ client；**claude/codex/gemini 现在原生讲 ACP**（recursive/aider/cursor/agy 未确认，大概率是各自私有格式）。
- **`_meta` 本质痛点**：当前每个 adapter 手写各自 `_meta` 提取——claude/cursor 解析 CLI 的 JSON（token/model）；gemini/codex/agy/aider 只拿 `model`（CLI 输出不暴露 token，无 token 统计）；recursive 最 bespoke：正则抠 `reason:BudgetExceeded`/`[done after N steps] reason:` + 退出码 101/≥128 判 panicked + 读 transcriptOut 文件数 messages。机制统一（boxed string + 不可枚举 `_meta`），内容逐 CLI 私有。
- **ACP 对 executor 的判断**：对 flowx 核心**比 agentproc 更契合**——ACP 把 token/model/tool-call/finish 作为**协议级结构化事件**携带（不再逐 CLI 抓），且 `session/request_permission` 双向权限天然映射到 flowx 的 hitl.js，`session/new`+resume 覆盖多轮。代价：ACP agent 是**有状态长驻子进程**（session 跨多 prompt），不同于 flowx 当前一次性 spawn 模型，需要把 executor 从"函数调用"改成"维持 session 的连接对象"；实现可用官方 `@agentclientprotocol/sdk`（TS）做 client 端，写一次覆盖所有 ACP agent。
- **修正后的三层建议**：① 原生 adapter 保留 recursive/aider/cursor/agy（需 bespoke 控制，尤其 recursive 的 budget/panic 检测）；② **ACP executor（优先加）** 吃 claude/codex/gemini，结构化 `_meta` + HITL 映射；③ agentproc executor（次要）吃长尾 dumb CLI。ACP 不是全量替换，而是"新增一种 executor 类型"。

## 产出：ACP Executor 落地方案（草案 v1）

已写入 `docs/exec-plans/active/acp-executor.md`，待讨论。

**核对到的关键协议事实（修正前两轮乐观判断）：**
- ACP 稳定提供 `usage_update`（session 级累计 `used`/`size`/`cost`）+ `stopReason` + `agent_message_chunk` 流式 + `tool_call`/`tool_call_update` 遥测 + `session/request_permission`（→HITL）+ `session/resume` + `session/close`（已稳定，前次 fetch 称"无 close"是过期信息）。
- ⚠️ **per-turn 的 input/output token 拆分是 DRAFT**（End-Turn Token Usage RFD 未稳定）。所以 ACP 当下白送的是 **session 级累计成本**，不是 per-turn 拆分——短期 ACP executor 在 token 维度比原生 recursive/claude adapter **少**（拿不到 per-turn input/output）。
- SDK 包名 `@agentclientprotocol/sdk` v1.2.0；flowx 走 **client** 角色：`client({name})` 注册 `requestPermission`/`sessionUpdate` handler → `connectWith(stream, ...)`。Gemini CLI 是官方生产级参考。

**方案要点：** 新增泛型 `acp` executor 类型，不替换 7 个原生 adapter。核心抽象 `AcpSession` 类（spawn agent 子进程 over stdio → session/new → prompt → 收集 update 累积 `_meta` → close）。与 resolveAgent/runAgent/runAgentChain/dry-run 的衔接点已列。`_meta` 映射表逐字段标注了"稳定/DRAFT"。

**7 个开放问题留给 Jeff 拍板**：① MVP 走无状态每次 spawn(A) 还是会话句柄复用(B，推荐 A)；② 接受 token 维度代价 or 先只迁不报 per-turn token 的 CLI；③ HITL 是否加 requestApproval 结构化语义；④ bin 路径安全（PATH/白名单目录/任意 bin）；⑤ 是否把 claude/codex/gemini 从原生迁到 acp（建议新增保留原生）；⑥ 长驻 session 与 concurrency 协调；⑦ ACP BYO-LLM 是否接受 provider。

## 配置：新增 minimax / deepseek 两个 agent（executor=claude）

- 在 `~/.flowcast/agents.json` 的 `agents` 下新增 `minimax`、`deepseek` 两个具名 agent：executor 均为 `claude`，provider 分别挂 `anthropic-minimax` / `anthropic-deepseek`（**不是** `minimax`/`deepseek` 原 key）。
- 原因：claude 执行器 `claudeProviderEnv` 把 provider 翻译成 `ANTHROPIC_BASE_URL`+`ANTHROPIC_AUTH_TOKEN`，claude code CLI 只能吃 anthropic 协议；`providers.json` 里 `minimax`/`deepseek` 是 openai 端点（`/v1`），只有 `anthropic-minimax`/`anthropic-deepseek` 是 anthropic 兼容端点，挂原 key 必失败。
- 前提：`MINIMAX_API_KEY` / `DEEPSEEK_API_KEY` 环境变量须存在（providers.json 用 `${...}` 插值）。
