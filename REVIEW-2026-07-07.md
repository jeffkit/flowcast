# flowcast 深度 Review（架构 + 代码实现）

> 评审日期：2026-07-07 · 视角：架构合理性 + 实现正确性 + 可维护性
> 结论先行：**架构水平高于多数同类项目**，依赖无环、续跑/原子写/路径安全做得到位、注释坦诚。
> 但 **L3 生成代码的信任边界是最脆弱的面**，且若干全局可变状态 + 一处 200+ 行的并发锁逻辑零单测。
> 以下按「必须修 / 建议修 / 可延后」分级。

---

## 一、总评亮点（先肯定，避免只挑刺）

- **依赖图是干净 DAG**：`executor.js ↔ agent.js` 旧循环依赖已通过把实现下沉到 `adapters.js` + `agent.js` 纯 re-export 解决（`agent.js` 仅 30 行）。当前分层 `叶子 → adapters/checkpoint/quality-gate → executor/loop/orchestrator` 无环。
- **路径安全贯穿全局**：`assertSafeIdent`（`helpers.js:19`）统一拦截拼进文件路径的标识符，`executor.js` 对 `systemPromptFile/transcriptOut/pricingFile/workspace/extraArgs` 做白名单 + `isSafePath` 校验，防 LLM 注入路径穿越。这是同类项目最容易翻车的地方，做得扎实。
- **持久化原子写**：`checkpoint.js` 用 write-rename（`checkpoint.js:431`）、大结果 sidecar + FNV-32 防碰撞、`sweepStaleTmp` 清孤儿，崩溃可恢复。
- **续跑锁设计认真**：`orchestrator/run.js` 用 `mkdir O_EXCL` 原子拿锁 + `owner.json` 记 PID/startedAt，并用 `ps -o etime=` 防 PID 复用竞态（`run.js:419`），1h stale 阈值给了一行推理注释。比「写 0-byte 文件」的旧方案稳妥。
- **dry-run 隔离**：`validate.js` 用独立临时 `HOME` 跑生成代码的干跑（`validate.js:207`），防止生成代码读 `~/.ssh`/`providers.json` 间接泄露。这个防御意识很好。
- **零运行时依赖 + 纯 ESM**：守得住，测试用 fake CLI / fake gate / temp git repo，不烧 API。

## 二、必须修（P0）

### P0-1 · L3 的「安全边界」靠正则白名单，且 `import * as fc` 已被承认可绕过

生成 flow 的信任边界依赖 `scanImports`（`validate.js:76`）的**正则 import 扫描** + 假 HOME 干跑。但代码自己承认了两个硬伤：

- `import * as fc from 'flowcast'; fc.spawnCapture(...)` 无法被静态拦截（`validate.js:27-28`、`validate.js:148` 注释明说）。而 `spawnCapture`/`spawnCli` 正躺在**公开** `flowcast` 表面（`index.js:11`）。也就是说一个半可信的 LLM 生成代码只要用命名空间导入，就能驱动任意子进程——白名单形同虚设。
- 干跑校验本身不防 `fc.spawnCapture`：干跑只是传 `FLOWCAST_DRY_RUN=1` 跑 flow，若 flow 不理这个 flag 直接调 `spawnCapture`，干跑阶段就会**真的执行**恶意子进程（假 HOME 只挡文件读，不挡网络/exec）。

**这关系到项目定位**：README 把「自改安全沙箱」「让 agent 安全地改自己的代码」当卖点，但真正的安全面（L3 生成 flow）却是 lint 级约束，不是真沙箱。要么在文档里把威胁模型讲清楚（只防「粗心 LLM」，不防「恶意 prompt」），要么用结构而非正则关掉这扇门。

**建议（结构化解法，推荐）**：新增 `flowcast/generated` 子路径导出，**只**导出 FLOW_API 白名单（Checkpoint/runAgent/runGate/loop/...），把 `spawnCapture`/`spawnCli` 等危险原语留在 `flowcast`（宿主/工具用）。生成 flow 的 import 白名单改为只允许 `flowcast/generated` + `util`，这样即便 `import * as fc from 'flowcast/generated'` 也拿不到危险符号。这是一次性改动，彻底消除命名空间绕过，且让「公开 API」语义自洽。

### P0-2 · 最复杂的并发锁逻辑零单测

`orchestrator/run.js` 的 `acquireLock`/`isPidAlive`/`isPidLockOwner`/`parseEtimeSec`（约 `run.js:320-448`，200+ 行）是全仓并发正确性最密集的代码，且 `test/` 里**没有 `orchestrator-run.test.js`**（现有 22 个测试文件，覆盖了 validate/generate/decompose/cli/codegen，唯独漏了锁）。PID 复用竞态、stale 判定、跨进程 mkdir 这些都是一踩一个坑的地方，没有单测等于裸奔。

**建议**：补 `orchestrator-run.test.js`，用注入的 `mkdirSync`/`process.kill`/`spawnSync` stub 模拟三种死路径：① 正常繁忙锁；② PID 已死（stale 清理）；③ PID 复用（新进程 etime < 锁 age 应判为非持有者）；④ crash 在 mkdir 与写 owner 之间（见 P1-3）。

## 三、建议修（P1）

### P1-1 · `makeAgentResult` 用 `new String()` 装箱，是个长期 footgun

`helpers.js:94-106` 把 agent 结果做成 `String` 对象（`typeof === 'object'`，`helpers.js:81` 注释自己也标了 ⚠️）。任何消费方写 `if (typeof r === 'string')` 都会**静默误判**。项目为此专门加了 `agentText`/`agentMeta` 帮手（`helpers.js:114-128`），等于用两个新 API 给一个坏设计打补丁。

兼容性矩阵里「`JSON.stringify(r)` 仍是纯字符串」的卖点，用 `toJSON()` 也能达到，且不必牺牲 `typeof`。

**建议**：改成普通对象 `{ text, _meta }` 或带 `toString()`/`toJSON()` 的类。`toJSON()` 返回 text 即可保持序列化兼容，`typeof` 也诚实。一次性把内部 `agentText`/`agentMeta` 改成直接读属性，调用点更直白。

### P1-2 · 全局可变单例并发不安全，靠注释警告而非结构约束

至少 4 处进程级全局单例：
- `hitl.js:143` `_hitlBackend`（HITL 后端）
- `executor.js:271` `_defaultCwd`（已 self-deprecate，但还在用）
- `adapters.js:42` `_agentEventSink`
- `memory.js:16-22` `_lineCountCache` / `_entriesCache`（per-process 无界，200 上限 FIFO）

在 `fanOut`/并发 `parallel` agent 调用里，若两个 flow 各自 `setHitlBackend` / `setAgentEventSink`，会互相污染。注释说「fanOut 子任务跑在独立子进程，不受影响」——但 `loop` + `parallel` 同进程内就受影响，且没有任何机制保证隔离。

**建议**：把 backend / sink / cwd 收进显式的 context 对象（`{ backend, sink, cwd }`）沿调用链传递，而不是模块级全局。最小改动：至少在 README/API 文档明确「这些 setter 是进程全局、非并发安全，并发场景请用子进程隔离或显式传参」。

### P1-3 · 锁逻辑有 1 小时不可用窗口（crash between mkdir 与 owner write）

`run.js:328-334`：先 `mkdirSync(lockDir)` 拿锁，再 `writeFileSync(owner.json)`。若进程在两者之间被 SIGKILL：
- 锁目录存在，但无 `owner.json`。
- 下一个进程在 `run.js:346-362` 等 `LOCK_WAIT_TRIES×100ms` 后，因 `owner.json` 缺失走 `statSync(lockDir).mtimeMs` 兜底，而 mtime 通常 < `STALE_LOCK_MS`（1h），于是抛 `LOCK_OWNER_PENDING`（`run.js:373`）。
- **后果**：该 `runId` 被锁死长达 1 小时，即使没有任何活进程持有它。注释把窗口说成「极短竞态」，但*后果*持续 1h——对重复跑/CI 场景是真实可用性问题。

**建议**：把「`lockDir` 存在但无 `owner.json` 且 mtime > 5s（而非 1h）」直接判 stale 清理；或先把 `owner.json` 写到临时名再 `rename` 进 `lockDir`，让「有锁必有 owner」始终成立。

### P1-4 · 双层回退链冗余（claude providerFallbacks vs runAgentChain）

- `adapters.js:108-128`：`claude` adapter 内部已实现 `provider → ...providerFallbacks` 回退链。
- `executor.js:410-465`：`runAgentChain` 又做了一层**跨 CLI** 回退（claude→cursor→...），带指数退避 + 持久化限流。

两层回退语义重叠，且日志格式各写一套（`adapters.js:121` vs `executor.js:454`），排查时容易混淆「到底是 provider 层退还是 CLI 层退」。

**建议**：明确分层职责——adapter 只管「单 provider 内多 key 回退」；`runAgentChain` 只管「跨 CLI 回退」。不要让 `claude` 自己再吃一层 fallback，避免嵌套。或者反过来，把 provider 层回退上提到 `runAgentChain` 统一编排。

### P1-5 · `runAgentChain` 里藏了对外 LLM 调用

`executor.js:441`：`recordRateLimit(..., { useLLM })` 当 `FLOWCAST_RATE_LIMIT_LLM` 置位时会调用外部 LLM 解析限流文案。这是「限流本地库」里的隐藏网络副作用 + 成本，且默认不开（靠 env 开关），文档未强调。

**建议**：保持默认本地解析；在 CLAUDE.md/README 显式标注「启用 `FLOWCAST_RATE_LIMIT_LLM` 会向外部 LLM 发送限流响应片段，产生额外 API 成本」。

## 四、可延后（P2）

- **统一 logger**：日志混用 `console.warn`/`console.error`/`console.log` 与 `makeEvent` 结构化事件（`helpers.js:50` 双 `event`/`type` 字段向后兼容）。长生命周期 daemon 场景下难以分级/重定向。建议抽象一个 `log(level, ...)` 出口，默认走 console，可注入。
- **锁里的 `spawnSync('ps')` 阻塞事件循环**：`run.js:425` 同步调用 `ps`，超时 2s，在 `fanOut` 多锁并发时会串行阻塞。可用非阻塞方式（如 `child_process.spawn` 异步 + 短超时），或缓存 etime。
- **memory 召回是裸子串打分**：`memory.js:81-91` `scoreEntry` 只做 `includes` + tag 命中。项目注释已声明「接口稳定，日后换向量召回」，可接受；但 `recall` 默认按时间倒序，无 query 时其实没用到打分，符合预期。
- **`concurrency.js` 的 `failFast` 在无并发上限时退化为 no-op**（注释已说明），文档化即可，误用风险低。
- **命名小瑕疵**：工作目录叫 `flowx`，但 `package.json`/`README`/`CLAUDE.md` 全用 `flowcast`。疑似改名中途，建议统一，避免 clone 后困惑。
- **`index.js` 平铺 L1/L2/L3**：公开 surface 同时导出 `spawnCapture`（L1 进程原语）、`runAgent`/`runGate`（L2）、`orchestrate`（经 `./orchestrator` 子路径，L3）。配合 P0-1 的 `flowcast/generated` 拆分，能天然把「宿主/工具用」与「生成代码用」两个 surface 分开。

## 六、一句话总结

> 骨架很结实，但「让 LLM 生成并执行代码」这条主路径的安全假设建立在**约定 + 正则**之上，而不是**模块边界 + 隔离**。先把 P0 两件事做了——结构化解开生成代码的导入面、给锁逻辑补单测——其余都是打磨。
